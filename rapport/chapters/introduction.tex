\chapter{Introduction}

\epigraph{\textit{``The world is built up of patterns, and so is the mind.''}}{--- Maxim Gumin}

\section{Contexte}

La \textbf{génération procédurale} est devenue centrale dans plusieurs domaines : jeux vidéo (Minecraft, No Man's Sky, Townscaper), simulation, graphisme, level design, art génératif. Le défi commun : produire des artefacts variés mais qui suivent une cohérence locale. La grille générée ne doit pas seulement être valide globalement, elle doit \textit{ressembler} localement à un échantillon de référence fourni par le designer.

Parmi les algorithmes répondant à ce besoin, le \textbf{Wave Function Collapse} (WFC), introduit par Maxim Gumin en 2016 \cite{gumin2016wfc}, occupe une place singulière. Inspiré par analogie avec la mécanique quantique (chaque cellule commence en superposition de toutes ses valeurs possibles, puis « se collapse » progressivement), WFC se distingue par sa simplicité conceptuelle : aucun apprentissage automatique, aucun réseau de neurones, juste une heuristique gloutonne combinant entropie minimale et propagation de contraintes.

\subsection{Le problème WFC en une phrase}

À partir d'un échantillon $S$ (grille d'entiers de taille modeste) et d'un paramètre $N$ (taille de tuile), produire une grille $G$ (taille libre) telle que tout sous-bloc $N \times N$ de $G$ apparaisse au moins une fois dans $S$. Si $S$ contient des immeubles avec fenêtres, $G$ produit un skyline cohérent ; si $S$ contient une plante avec fleurs, $G$ produit un jardin de plantes similaires (figure~\ref{fig:wfc_paper_outputs}).

\subsection{Intérêt du calcul haute performance}

WFC paraît simple mais sa complexité est conséquente. La phase de \textit{sélection min-entropie} re-scanne la wave entière à chaque collapse, soit $O(\text{rows} \cdot \text{cols} \cdot L)$ pour chaque itération, multiplié par $\sim \text{rows} \cdot \text{cols}$ collapses au total. Sur une grille $256 \times 256$ avec $L = 11$ tuiles, cela représente environ $4.6 \cdot 10^9$ opérations de scan dans la version naïve.

La \textbf{propagation BFS} ajoute un second coût : chaque collapse déclenche une cascade qui peut toucher des milliers de cellules avant de stabiliser. Cette propagation est intrinsèquement séquentielle au sein d'un niveau BFS donné, mais les niveaux successifs offrent du parallélisme exploitables.

Le \textbf{calcul haute performance} (HPC) répond à ce défi en exploitant :
\begin{itemize}
    \item le \textbf{parallélisme à mémoire partagée} via OpenMP, en particulier l'API \texttt{\#pragma omp task} explicite recommandée par le sujet ;
    \item la \textbf{portabilité GPU} via Kokkos \cite{kokkos2014}, framework de l'écosystème DOE qui compile le même code source vers OpenMP, CUDA, HIP, ou SYCL ;
    \item les \textbf{optimisations bas niveau} : bitsets packés sur \texttt{uint64\_t}, atomics relaxées, alignement cache, NUMA first-touch.
\end{itemize}

\section{Objectifs du projet}

Le projet poursuit cinq objectifs articulés autour de trois axes principaux.

\subsection{Implémenter trois backends WFC}

Le sujet demande explicitement (i) une version série de référence, (ii) une version parallèle utilisant l'API \texttt{\#pragma omp task} explicite \textbf{ou} Kokkos, et (iii) une extension multi-valeurs. Nous avons choisi d'implémenter \textbf{les trois backends} (série, OpenMP-tasks, Kokkos) afin d'analyser comparativement les paradigmes de parallélisation. Les trois doivent produire un \textbf{output bit-identique} pour un même seed, ce qui impose des contraintes fortes sur le tirage aléatoire et la réduction parallèle.

\subsection{Analyser la performance et la scalabilité}

Le second objectif consiste à évaluer rigoureusement les performances :
\begin{itemize}
    \item Mesures de temps d'exécution sur configurations variées (grilles de $32 \times 32$ à $512 \times 512$, $L$ de 11 à 73 tuiles) ;
    \item Métriques de scalabilité : speedup, efficacité, fit Amdahl ;
    \item Expérimentations sur le supercalculateur \textbf{Romeo} (Université de Reims Champagne-Ardenne) avec configurations jusqu'à 192 threads OpenMP et un GPU NVIDIA GH200 ;
    \item Analyse du \textit{varying input dataset} : impact du sample (binaire vs multi-valeurs), du paramètre $N$, et de la taille de sortie.
\end{itemize}

\subsection{Livrer des extensions opt-in}

Au-delà du sujet, nous avons livré quatre extensions optionnelles, toutes activables sans dégrader le chemin par défaut :
\begin{enumerate}
    \item \textbf{Parallel attempts} (\texttt{--parallel-attempts K}) : lance K attempts WFC indépendants en parallèle, garde le succès d'index minimum (déterministe, +2.14× wallclock sur \texttt{terrain N=3}) ;
    \item \textbf{Symétries D4} (\texttt{--symmetries S}) : étend le tile set par rotations + réflexions (zéro coût si désactivé) ;
    \item \textbf{Backtracking} (\texttt{--backtrack}) : alternative au restart-on-contradiction, avec snapshot delta-encodé pour minimiser la mémoire ;
    \item \textbf{Démo Unreal Engine 5} (\texttt{-DBUILD\_DUNGEON=ON}) : CLI \texttt{wfc\_dungeon} qui émet un JSON consommé par un plugin UE 5.7 spawnant des meshes en 3D.
\end{enumerate}

\section{Contributions et organisation du rapport}

\subsection{Contributions}

Les principales contributions de ce travail sont :

\begin{enumerate}
    \item \textbf{Trois backends bit-identiques} (serial, OpenMP, Kokkos) avec déterminisme garanti par hash SplitMix64 sur \texttt{(cell\_id, seed)} et réduction min-entropie en ordre de chunk fixe ;

    \item \textbf{Refactor Kokkos GPU-portable} : le même code source compile et tourne sur CPU OpenMP et GPU CUDA grâce à une branche \texttt{if constexpr (kHostOnly)} qui choisit zero-copy (CPU) ou \texttt{deep\_copy} (GPU) ;

    \item \textbf{Optimisations OMP A/B-testées} : frontier-threshold (+25\% à 64 threads), relaxed-load atomique (-50\% trafic), LTO (+3.4\%). Plusieurs candidats rejetés (log-cache thread-local, vector FIFO) pour cause de régression ;

    \item \textbf{Trois extensions algorithmiques} (parallel attempts, symétries D4, backtracking) intégrées de façon strictement opt-in : zéro impact mesurable sur le chemin par défaut, vérifié par perf sanity check ;

    \item \textbf{Validation expérimentale étendue} : 220+ runs benchmarkés sur Romeo (jobs SLURM 543692, 544061, 544356, 544361), 99\% de couverture de lignes (gcovr), 24 980 checks d'invariants à travers 15 suites de tests.
\end{enumerate}

\subsection{Organisation du rapport}

Ce rapport est structuré en douze chapitres, suivis d'annexes techniques :

\begin{itemize}
    \item Le \textbf{Chapitre 2} présente les \textit{préliminaires} : complexité, paradigmes de parallélisation (OpenMP, Kokkos), métriques HPC ;

    \item Le \textbf{Chapitre 3} formalise le \textit{problème WFC overlapping} : définitions, propriétés, applications ;

    \item Le \textbf{Chapitre 4} détaille l'\textit{approche algorithmique séquentielle} : extraction toroïdale, compatibilités, entropie de Shannon pondérée, BFS de propagation ;

    \item Le \textbf{Chapitre 5} décrit l'\textit{implémentation C++} et l'ingénierie logicielle : Bitset packé, Wave NUMA-friendly, organisation en libs CMake ;

    \item Le \textbf{Chapitre 6} présente la \textit{parallélisation OpenMP} avec \texttt{\#pragma omp task} explicite et les optimisations gardées ou rejetées en A/B ;

    \item Le \textbf{Chapitre 7} traite de la \textit{parallélisation Kokkos} et du refactor GPU-portable, avec les contraintes spécifiques au backend CUDA ;

    \item Le \textbf{Chapitre 8} décrit les \textit{extensions algorithmiques} (parallel attempts, symétries D4, backtracking) et leur stratégie d'intégration opt-in ;

    \item Le \textbf{Chapitre 9} décrit le \textit{protocole expérimental} et la méthodologie de benchmarking sur Romeo ;

    \item Le \textbf{Chapitre 10} présente les \textit{résultats} et leur analyse : speedup et efficacité, fit Amdahl, varying datasets et configurations ;

    \item Le \textbf{Chapitre 11} discute des \textit{limites et perspectives} : NUMA partitioning, batch GPU, priority queue incrémentale, SIMD bitset ;

    \item Le \textbf{Chapitre 12} \textit{conclut} en résumant les apports, les leçons HPC retenues, et les recommandations pour une suite éventuelle.
\end{itemize}

Les \textbf{Annexes} fournissent les détails techniques : commandes de compilation, format CSV des benchmarks, descriptions des samples, et table des optimisations testées.
